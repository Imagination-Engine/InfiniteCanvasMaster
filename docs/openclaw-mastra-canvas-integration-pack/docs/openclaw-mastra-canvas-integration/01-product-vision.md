# 01 — Product Vision

## One-Line Definition

An **OpenClaw Block** is a live agent/runtime cell inside the Balnce Imagination Canvas, orchestrated by Mastra, governed by Balnce policy, and able to perform scoped tasks through OpenClaw tools, skills, sessions, channels, models, and runtime capabilities.

## It Is Not

The OpenClaw Block is not:

- a chatbot card
- a generic automation card
- an iframe around the OpenClaw UI
- a raw terminal wrapper
- a free-running local agent
- an unrestricted host process
- a visual-only placeholder
- a toy demo of agent orchestration
- a card with fake status labels

## It Is

The OpenClaw Block is:

- a managed runtime binding
- a permissioned execution cell
- a sandboxed tool surface
- a visual task cell
- a Mastra-invokable capability
- a canvas-native agent block
- an inspectable work unit
- a stoppable/pauseable execution surface
- a provenance-emitting artifact producer
- a participant in group orchestration
- a possible bridge between the user’s local machines, device mesh, Edge Twin, and cloud models

## Why This Matters

The Imagination Canvas is not simply a place where users put notes. It is an operating surface for thought, intent, artifacts, memory, agents, goals, workflows, and applications.

OpenClaw adds a powerful execution layer because it can represent a self-hosted assistant runtime, skill environment, device/channel bridge, and tool-using agent system. Mastra adds the higher-level substrate to route, supervise, sequence, checkpoint, evaluate, observe, and coordinate these blocks.

The result is a canvas where a user can create one OpenClaw block for a focused task, or place twenty OpenClaw blocks into a coordinated group and ask Mastra to orchestrate them as a task force.

## Product Experience Target

The user should be able to:

1. Add an OpenClaw Block to the canvas.
2. Choose or bind a runtime/session/workspace.
3. Give it a task.
4. See what it is allowed to do.
5. See what model/compute route it will use.
6. Watch it work.
7. Approve sensitive actions.
8. Pause or stop it.
9. Inspect the event timeline.
10. Convert outputs into artifacts, notes, workflows, goals, offers, or app blocks.
11. Group multiple OpenClaw Blocks and ask them to work together.
12. Trust that permissions, budgets, provenance, memory, and identity boundaries are not being silently bypassed.

## Balnce-Native Differentiation

The OpenClaw Block must become part of Balnce’s deeper primitives:

- **Intent**: the block can be assigned user intent.
- **Memory**: the block operates within explicit memory scope.
- **PLOG / Provenance**: outputs and sensitive actions emit provenance.
- **AURA**: runtime spend, rewards, or cost budgets can be represented.
- **VLAD / Identity**: agents and actions should bind to identity/provenance where the app supports it.
- **Edge Twin**: long-running or high-compute jobs can be delegated.
- **Device Mesh**: local devices can pool inference or execution.
- **Human Checkpoints**: sensitive operations require approval.
- **Canvas Blocks**: outputs become tangible blocks on the canvas.

## Success Condition

The user does not experience “twenty chatbots.”
The user experiences a spatial command center of specialized, trustworthy, inspectable, working agent cells.
